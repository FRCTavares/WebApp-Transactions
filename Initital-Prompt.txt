F - TRANSACTIONS WEBAPP PROJECT PROMPT AND BUILD PLAN

Project name:
F - Transactions

Project goal:
Build a personal finance transactions webapp that I can use on my Mac and on my iPhone to store, import, clean, categorise, and analyse my bank transactions.

The app should let me understand where my money comes from, where it goes, and who owes me money.

Current priority:
Backend and data logic first.
Frontend should stay simple, ugly, and functional until the backend, database schema, import logic, deduplication, and analysis endpoints are reliable.

I do not want to waste time making the frontend pretty before the logic works properly.


============================================================
1. USAGE AND DEPLOYMENT STRATEGY
============================================================

Current usage:
- Local-first app.
- Runs on my Mac.
- Built from iTerm2 on macOS.
- Should eventually be usable from my iPhone, at least over the same local network.

Current local stack:
- Backend: FastAPI
- Database: SQLite
- Frontend: React + TypeScript + Vite
- Styling: minimal Tailwind CSS

Future hosting possibility:
Later, I may host the project using:
- Vercel for the frontend
- Supabase for database/auth/storage if needed
- Render for the backend if needed

Important future-proofing rule:
For now, use SQLite.
But design the backend so SQLite can later be replaced by Supabase/Postgres without rewriting the whole app.

This means:
- Keep database access isolated.
- Do not scatter raw SQL everywhere.
- Use repository/service layers.
- Keep business logic separate from API routes.
- Keep import logic separate from database logic.
- Keep frontend API calls separate from React components.

Do not add Supabase, Vercel, Render, auth, or deployment logic yet.
Just avoid architecture choices that would block them later.


============================================================
2. CORE DATA SOURCES
============================================================

There are two ways of adding transaction data:

1. Manual entry
   - Add transactions manually through a simple form/table.

2. CSV imports
   - Import CSVs from:
     - ActivoBank
     - Revolut
     - Trading 212

The CSV import system must be modular and source-specific.

There should be one importer per source:
- activobank.py
- revolut.py
- trading212.py

Each importer should convert source-specific CSV rows into one shared internal transaction format.

============================================================
4. IMPORTANT ARCHITECTURE DECISION
============================================================

Do not create three completely separate transaction systems if that makes the logic messy.

Preferred internal structure:
- Use one clean transactions model/table for Money In and Money Out.
- Use a direction field to separate income from expenses:
  - direction = "in"
  - direction = "out"

Then expose separate frontend views:
- Money In view filters transactions where direction = "in"
- Money Out view filters transactions where direction = "out"

Use a separate owed_items model/table for Money Owed To Me.

This keeps analytics, imports, deduplication, filters, and summaries simpler.

5. CORE DATABASE TABLES

------------------------------------------------------------
5.1 transactions table
------------------------------------------------------------

Purpose:
Stores all real money-in and money-out transactions.

Suggested fields:
- id
- date
- description
- raw_description
- amount
- direction
- source
- account
- category
- subcategory
- currency
- merchant
- notes
- import_batch_id
- external_id
- dedupe_hash
- created_at
- updated_at

Field explanations:
- id:
  Internal transaction ID.

- date:
  Transaction date.

- description:
  Cleaned transaction description.

- raw_description:
  Original bank/app description from CSV or manual entry.

- amount:
  Positive numeric value.
  Direction is controlled by the direction field, not by negative amounts.

- direction:
  Either "in" or "out".

- source:
  Where the transaction came from.
  Examples:
  - manual
  - activobank
  - revolut
  - trading212

- account:
  Optional account name.
  Examples:
  - ActivoBank current account
  - Revolut
  - Trading 212

- category:
  Main category.
  Examples:
  - Salary
  - Groceries
  - Restaurants
  - Transport
  - Investing

- subcategory:
  Optional more specific category.

- currency:
  EUR, USD, GBP, etc.

- merchant:
  Optional cleaned merchant name.

- notes:
  Manual notes.

- import_batch_id:
  Links transaction to the CSV import batch that created it.

- external_id:
  Source-provided transaction ID, if available.

- dedupe_hash:
  Stable hash used to avoid duplicate imports.

- created_at:
  Creation timestamp.

- updated_at:
  Last update timestamp.


------------------------------------------------------------
5.2 owed_items table
------------------------------------------------------------

Purpose:
Stores money that other people owe me.

Suggested fields:
- id
- person
- amount_total
- amount_paid
- amount_remaining
- reason
- status
- due_date
- linked_transaction_id
- notes
- created_at
- updated_at

Field explanations:
- id:
  Internal owed item ID.

- person:
  Person who owes me money.

- amount_total:
  Original total amount owed.

- amount_paid:
  Amount already paid.

- amount_remaining:
  Can be computed as amount_total - amount_paid.

- reason:
  Why they owe me money.
  Examples:
  - Dinner
  - Trip
  - Tickets
  - Rent split
  - Groceries

- status:
  Suggested values:
  - open
  - partially_paid
  - paid
  - cancelled

- due_date:
  Optional due date.

- linked_transaction_id:
  Optional link to a transaction, usually a Money Out transaction.

- notes:
  Extra notes.

- created_at:
  Creation timestamp.

- updated_at:
  Last update timestamp.


------------------------------------------------------------
5.3 import_batches table
------------------------------------------------------------

Purpose:
Tracks each CSV import.

Suggested fields:
- id
- source
- filename
- imported_at
- rows_total
- rows_inserted
- rows_skipped
- status

Field explanations:
- id:
  Import batch ID.

- source:
  CSV source.
  Examples:
  - activobank
  - revolut
  - trading212

- filename:
  Original CSV filename.

- imported_at:
  Timestamp.

- rows_total:
  Number of rows read.

- rows_inserted:
  Number of new transactions inserted.

- rows_skipped:
  Number of duplicate/invalid rows skipped.

- status:
  Suggested values:
  - success
  - partial
  - failed


------------------------------------------------------------
5.4 category_rules table
------------------------------------------------------------

Purpose:
Optional table for basic rule-based categorisation.

Suggested fields:
- id
- pattern
- category
- subcategory
- direction
- source
- priority
- created_at
- updated_at

Example rules:
- If description contains "UBER" -> Transport
- If description contains "PINGO DOCE" -> Groceries

- If description contains "SALARY" -> Salary

============================================================
7. CSV IMPORT ARCHITECTURE
============================================================

The import system must have two stages:

1. Preview
2. Commit

Do not import directly into the database without preview.

Correct flow:
Upload CSV
-> choose source
-> parse rows
-> normalise rows
-> validate rows
-> compute dedupe hashes
-> detect duplicates
-> show preview
-> show duplicates/skipped rows
-> user confirms
-> commit to database


------------------------------------------------------------
7.1 Importer modules
------------------------------------------------------------

Create one importer per source:

backend/app/importers/base.py
backend/app/importers/activobank.py
backend/app/importers/revolut.py
backend/app/importers/trading212.py

Each source importer should convert CSV rows to a shared internal format.

Example shared structure:

NormalisedTransaction:
- date
- raw_description
- description
- amount
- direction
- source
- account
- currency
- external_id
- notes


------------------------------------------------------------
7.2 Shared import pipeline
------------------------------------------------------------

All source importers should feed into one shared pipeline:

CSV row
-> source parser
-> normalised transaction
-> validation
-> dedupe hash
-> category guess
-> duplicate detection
-> preview
-> commit

The importer should only know how to read that source's CSV format.
It should not know about database writes, frontend views, or summaries.


------------------------------------------------------------
7.3 Deduplication
------------------------------------------------------------

Use a stable dedupe hash.

Suggested initial hash:

source + date + amount + direction + raw_description + currency

Example logic:
dedupe_hash = sha256(
    f"{source}|{date}|{amount}|{direction}|{raw_description}|{currency}"
)

Later, if a source provides a stable transaction ID, include or prefer that.

Deduplication goal:
Importing the same CSV twice should not create duplicate transactions.

============================================================
15. MODULARITY RULES
============================================================

No file should exceed 1000 lines.

Prefer smaller focused files over large files.

Do not create duplicate:
- API endpoints
- React components
- models
- schemas
- importers
- utility functions
- database access functions

Before adding a new file or feature:
- Check whether an existing module should be extended.
- Check whether similar logic already exists.
- Avoid parallel systems.

Backend modularity:
- Routers handle HTTP only.
- Services handle business logic.
- Repositories handle database access.
- Importers handle CSV source parsing.
- Utils handle generic helpers.
- Schemas handle API validation/serialization.
- Models handle database structure.

Frontend modularity:
- Pages compose screens.
- Components display reusable UI.
- API modules call backend endpoints.
- Types define API/data shapes.
- Utils format dates, money, etc.

Do not put large logic directly inside React components.
Do not put database queries directly inside API routers.
Do not put CSV parsing directly inside endpoints.

- create()
- list()
- get_by_id()
- update()
- delete()

ImportBatchRepository:
- create()
- update_status()
- get_by_id()

CategoryRuleRepository:
- create()
- list()
- update()
- delete()

Services should depend on repositories.
Routers should depend on services.

This makes future migration easier.
